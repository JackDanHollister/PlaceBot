PlaceBot

AI-assisted georeferencing of verbatim locality data.

Turn free-text locality descriptions — like those on natural-history specimen labels — into standardised decimal coordinates, an uncertainty radius, and a full geographic hierarchy. Open source, vendor-agnostic, and built to run at scale.

Why PlaceBot exists

Hundreds of millions of biodiversity occurrence records carry only a free-text description of where they were collected, not the coordinates that research, distribution modelling and conservation policy depend on. These records are historically rich but spatially unusable at scale, and the backlog grows faster than institutions can clear it.

Manual georeferencing is accurate, but it is slow, costly, and inconsistent between people. PlaceBot tackles this bottleneck directly: it reads the descriptive text and returns standardised coordinates with an explicit, machine-readable estimate of how uncertain they are.

Crucially, PlaceBot does not treat an AI coordinate as a final answer. It wraps large language models in a deterministic, auditable workflow and flags exactly which records a human should check — turning georeferencing from a per-record manual task into a scalable, cost-bounded, partly automated one.

What it does

A deterministic, auditable workflow around the model — not a black box.

Vendor-agnostic

Use Anthropic, OpenAI or Google models via API, OpenRouter for one-key multi-vendor routing, or local Ollama/Qwen models for fully offline, privacy-sensitive processing.

Deterministic pre-screening

Before any model call, PlaceBot detects existing coordinates, UK Ordnance Survey grid references, UTM and DMS values in Python and converts them — the model only interprets the residual descriptive text.

Structured, transparent output

Every record returns coordinates, a coordinate-source classification, categorical confidence, an uncertainty radius, a full geographic hierarchy, and processing notes documenting the reasoning.

Cost control

Prompt caching (up to ~90% savings) and asynchronous batch APIs (a further ~50%), with staggered batching and multi-key support for very large jobs.

Native GBIF & Darwin Core I/O

Reads occurrence exports and Darwin Core files as-is — recognising terms like verbatimLocality and decimalLatitude — and can emit Darwin Core output. CSV, TSV, JSON and GeoJSON supported.

Dedup & reconstitute

placebot-prep georeferences each unique locality once; placebot-expand re-attaches the result to every record that shared it. You pay for and review one candidate per distinct place.

Ensemble triage

Run two strong models and let placebot-ensemble tag each record with an agreement category and the distance between estimates — so curators can filter straight to the records that need a human.

Works offline

With Ollama and a local model, the entire workflow runs on your machine with zero API spend — useful for sensitive data.

Quick start

Using the graphical interface — the easiest way in.

  1. Install

    Run pip install placebot, or download a desktop installer above.

  2. Launch

    Run placebot-gui (or double-click the shortcut). PlaceBot opens in your web browser.

  3. Add a key

    Paste an API key into the sidebar (Anthropic, OpenAI, Google or OpenRouter). Keys are session-only unless you tick Remember on this computer. To run offline, install Ollama, pull a local model, and pick it under Model source → Local Ollama.

  4. Upload your data

    Upload a CSV or TSV with at least a record-identifier column and a locality-text column. A country column and any existing coordinates are optional but improve results. GBIF exports and Darwin Core files work as-is — no renaming.

  5. Choose a model

    Pick one from the comparison table. High-tier Gemini and GPT models gave the best accuracy in our benchmarks; low-tier models are cheaper and faster.

  6. Run

    Choose real-time (small jobs), batch (large jobs, ~50% cheaper) or staggered batch (very large jobs). Deduplication is on by default, so each unique place is processed once.

  7. Download results

    Results save to your output folder with one-click access. Tick Use Darwin Core (DwC) output terms for columns like decimalLatitude and coordinateUncertaintyInMeters ready for re-publishing.

Recommended quality-control step. Run the same dataset through two strong models, then run placebot-ensemble (also on the GUI's Ensemble analysis page). PlaceBot tags each record with an agreement category and the distance between estimates. Accept records where the models agree closely; send disagreements for expert review.

How reliable is it?

PlaceBot ships with a reproducible evaluation: 12 models from four providers, benchmarked across two contrasting datasets — UK Bombus localities and globally distributed Odonata type-specimen localities — against manually curated reference coordinates. The benchmark datasets and archived model-comparison outputs are included in the repository so anyone can reproduce or extend the analysis.

1.61 km median error of the strongest model on the UK dataset (80% within 5 km)
5.83 km median error on the harder, globally heterogeneous dataset
2 models agreement between strong models is a robust reliability signal for triage

Three findings shape responsible use. Performance varies sharply by model and by locality complexity, so LLM coordinates should be treated as probabilistic, not authoritative. Where two strong models converge, records are far more likely to be close to truth — enabling a tiered workflow where high-concordance outputs are accepted and discordant cases go to expert review. And notably, reassessing divergent cases showed AI estimates sometimes corrected errors in existing manual georeferences — so PlaceBot can also quality-check legacy coordinates already in circulation.

PlaceBot complements careful single-record tools such as GBIF's GeoPick: it triages and pre-processes large backlogs and flags exactly which records warrant the human attention those tools provide.
